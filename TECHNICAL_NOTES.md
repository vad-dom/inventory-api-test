# Пояснение по техническим решениям

## Задание

Разработать REST API backend-сервера для складского учета товаров и остатков.

## Стек

* PHP 8.4
* Laravel 13
* MySQL
* Redis
* Nginx + PHP-FPM
* Docker Compose
* OpenAPI / Swagger
* PHPUnit

## Основные сущности

* `products` — товары
* `warehouses` — склады
* `stock_balances` — остатки товаров на складах
* `stock_movements` — история складских операций

## Принятые технические решения

### Обеспечение консистентности остатков

#### Решение:

* Для перемещений между складами используется пессимистическая блокировка `lockForUpdate()`;
* Для прихода и списания используется атомарное условное изменение остатка одним SQL-запросом.

#### Причина:

Перемещение одновременно изменяет остатки двух складов и требует согласованного изменения обеих записей при конкурентном доступе.
Приход и списание работают только с одним остатком, поэтому атомарного обновления достаточно и оно создает меньшую нагрузку на базу данных.

### Текстовый поиск по нескольким полям

#### Решение:

Для поиска используется оператор `LIKE`, а не FULLTEXT или ElasticSearch и подобные движки.

#### Причина:

Для production-проекта с большим объемом данных с большой вероятностью потребуется FULLTEXT или Elasticsearch.
Но FULLTEXT меняет поведение поиска, поэтому нужны более точные требования к поиску.
`LIKE '%...%'` гарантированно ищет подстроку в любом месте поля.
Поиск ведется только по справочникам `products` и `warehouses`, где вряд ли будет огромный объем данных.
Поэтому для простоты и предсказуемой семантики поиска выбран `LIKE`.

### Использование Route Model Binding

#### Решение:

Route Model Binding используется только для маршрута:
```http
GET /api/v1/stock/balances/{product}/{warehouse}
```

В остальных маршрутах используются параметры `{id}`.

#### Причина:

В задании для остальных эндпойнтов явно указаны параметры `{id}`.

### Индексы в базе данных

#### Решение:

Добавлены дополнительные индексы для полей `deleted_at` и `is_active`.

#### Причина:

Эти поля наиболее часто участвуют в фильтрациях и редко меняются.

### Индекс по `quantity`

#### Решение:

Индекс по полю `quantity` не создавался, хотя поле может часто участвовать в фильтрациях и сортировках.

#### Причина:

Поле часто изменяется при складских операциях, а поддержка индекса увеличила бы стоимость записи.

### Сортировка списков

#### Решение:

Сортировка по умолчанию реализована только для списка товаров.
Для остальных списков сортировка выполняется только при явной передаче параметров `sort` и `direction`.
При этом проверяется, чтобы `sort` и `direction` передавались только вместе.

#### Причина:

В задании сортировка по умолчанию указана только для списка товаров. 
Для остальных эндпойнтов не добавлено поведение, которое явно не описано.
Хотя для пагинации лучше, чтобы сортировка выполнялась всегда.

### Фильтрация по диапазону дат

#### Решение:

При фильтрации по датам используются `startOfDay()` и `endOfDay()`, а не SQL-функция `DATE()`.

#### Причина:

Использование функции `DATE(created_at)` не позволяет эффективно использовать индекс по полю `created_at`. 
Фильтрация диапазоном сохраняет возможность использования индекса.

### Soft Delete в статистике

#### Решение:

При расчете статистики не выполняется дополнительное исключение soft-deleted товаров и складов.

#### Причина:

При корректной работе системы товар или склад не могут быть удалены при наличии остатков/движений. 
Поэтому такие записи не попадут в статистику даже без дополнительной фильтрации. 

### Инвалидация кэша

#### Решение:

Инвалидация кэша статистики централизована через события моделей (Observers).

#### Причина:

Это позволяет автоматически очищать кэш независимо от того, где произошло изменение данных.

### `by_warehouse` в статистике

#### Решение:

В `by_warehouse` включаются только склады с положительным остатком.

#### Причина:

В задании не указано, нужно ли включать пустые склады.

### База данных для тестов

#### Решение:

Отдельная изолированная база данных для тестов не создавалась.

#### Причина:

Для тестового задания это привело бы к избыточному усложнению архитектуры без практической пользы.
Другой вариант был - database.sqlite, но лучше прогонять тесты на схожей инфраструктуре.

### Использование фиксированных данных в Seeders

#### Решение:

В Seeders используются фиксированные данные вместо Factory и Faker.

#### Причина:

Фиксированные данные дают предсказуемые идентификаторы и значения. 
Это удобно при подготовке документации, примеров OpenAPI и ручной проверке API.

### Swagger UI

#### Решение:

В проект интегрирован Swagger UI, хотя по заданию это необязательно.

#### Причина:

Это упрощает ручную проверку API и позволяет выполнять запросы без использования сторонних инструментов.

### Интерпретация `positions_with_stock`

#### Решение:

`positions_with_stock` трактуется как количество уникальных товаров, имеющих положительный остаток.

#### Причина:

В задании не уточняется, что именно считать позициями. 
Выбран вариант, который отражает количество уникальных товарных позиций, а не количество строк остатков по складам.

### Интерпретация `products_total` и `warehouses_total`

#### Решение:

В статистике в `products_total` и `warehouses_total` учитываются и активные, и неактивные записи.

#### Причина:

В задании не указано, что необходимо учитывать только активные записи.
